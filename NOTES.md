Final submission is a 4-layer, 6-head, 192-dim GPT with RMSNorm, SwiGLU MLPs, tied input/output embeddings, and RoPE instead of learned positional embeddings, at 1,946,496 params (97.3% of the 2M cap). Tokenization is a character+byte-fallback scheme (vocab 913: 256 raw-byte IDs plus 657 frequent characters learned from the training corpus), with a lossless decode(encode(x)) == x round-trip as required by evaluate.py. Training ran the full 2,000-step cap with micro-batch 8 and 2 gradient-accumulation steps (effective batch 16, block size 256), AdamW with decoupled weight decay, gradient clipping at 1.0, and a OneCycleLR schedule tuned to peak and anneal within exactly 2,000 steps. Final train loss was 1.4286 nats/token; measured dev bpb was 1.801 on dev_eval.txt, which is the only metric that's actually comparable across tokenizer choices. RoPE plus the larger SwiGLU width is the main lever versus a smaller baseline architecture: it uses far more of the 2M-param budget productively instead of leaving it on the table. The real cost is wall-clock: this configuration takes ~31 minutes for 2,000 steps on CPU, much slower than the ~3 min/run pace the assignment expects for iteration, so it was only run once at final scale rather than swept. A second, unresolved issue: this tokenizer learned fewer characters (657) than an earlier smaller-model run (768), so it produces noticeably more tokens per byte (1.28 vs 2.78 bytes/token) — likely under-serving the Hindi portion of the corpus — which partially works against the point of widening the context window. If iterating further, the first thing to try is not more architecture changes but a better tokenizer (raise max_chars, or move to real BPE merges) paired with this same model, since bpb is bounded as much by tokenization efficiency as by architecture here. No pretrained weights, external data, or non-stdlib/PyTorch/numpy dependencies were used anywhere in this run.
